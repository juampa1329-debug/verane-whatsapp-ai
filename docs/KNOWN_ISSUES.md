# KNOWN_ISSUES

Scope: SaaS only. These are risks observed from repository structure/code, not confirmed production incidents.

## Critical

- `saas-version/docker-compose.saas.yml` has a local default `SAAS_JWT_SECRET=change-me-local-saas-secret`. Production must override it.
- `saas-version/keys/saasprivate.key` exists. Treat it as secret material; do not expose, copy, or commit elsewhere without explicit verification.
- API embedded worker and standalone `worker` service can both run. Queue processors must remain idempotent and concurrency-safe.
- Verified production incident on 2026-05-28: tenant login 500 was caused by DB schema drift where `saas_users.locked_until` and `saas_billing_subscriptions.payment_failed_notice_sent_at` were missing. Production also showed missing `saas_billing_invoices`, confirming broader billing migration drift. Migration `069_saas_auth_billing_schema_drift_repair.sql` repairs this permanently.
- Verified follow-up production incident on 2026-05-28: after login, `/conversations`, `/dashboard/overview`, `/integrations`, `/advisor/briefing`, and `/auth/register` can return 500 when production DB is missing later CRM, verticalization, campaign or Intelligence schema. Migration `070_saas_crm_intelligence_schema_drift_repair.sql` repairs the affected app-boot and registration schema.
- Verified second follow-up on 2026-05-28: production can still 500 after `070` if older base Inbox/advisor/campaign columns from migrations `006`, `008`, `032`, `037`, vertical-pack seed columns/indexes, audit events or Advisor runtime tables are missing. Migration `071_saas_app_boot_schema_drift_repair.sql` repairs base conversation/message/outbound fields, vertical pack seed support, campaign preflight/A-B support and Advisor insight/recommendation tables.
- Registration now keeps tenant/trial creation mandatory but treats vertical pack seeding as non-critical inside a savepoint. If vertical pack seed fails because of schema drift or seed data issues, registration can succeed and records `auth.register` warning `vertical_pack_apply_failed`; operators must still apply the repair migrations and reapply the industry pack later.
- Verified production incident on 2026-05-28: Inbox Phase 24 read endpoint `/agents/multimodal-memory/events` can deadlock when runtime ensure helpers execute `ALTER TABLE` during concurrent browser reads. The read path now avoids runtime DDL and returns empty optional data when the Phase 24 table/columns are missing or incomplete; keep future frequently-polled read endpoints free of schema repair/DDL.
- Verified production risk on 2026-05-28: WhatsApp inbound can appear "stopped" if Meta still posts to a stale `/webhooks/{provider}/{endpoint_key}` URL after endpoint rows/keys changed. WhatsApp POST fallback now resolves tenant by WABA/Phone Number ID from the payload when an active connected WhatsApp integration and active webhook endpoint exist. A no-key compatibility route also accepts `/webhooks/whatsapp` and `/webhooks/meta` for legacy callbacks. This does not fix missing `subscribed_apps`, missing active endpoints, invalid tokens, wrong subscribed fields, or Meta not calling the Scentra API domain at all.
- Verified production deployment issue on 2026-05-28: the running API container was legacy/non-SaaS (`uvicorn app.main:app`) and had no `/app/app_saas` or `/app/migrations`. Follow-up Coolify logs showed the connected repo is `juampa1329-debug/Scentra-AI`, where SaaS files are already at repo root; for that app use base directory `/` plus Dockerfile `/backend/Dockerfile`. Only use `/saas-version` as base directory when Coolify imports the monorepo `verane-whatsapp-ai`.
- Verified production startup blocker on 2026-05-28: after deploying the correct SaaS image, migrations were applied through `072` but `schema_check` failed because `saas_billing_invoices.amount_cents` was missing. Migration `073_saas_billing_invoice_amount_cents_repair.sql` repairs the compatibility column. Until applied, the API container restarts and browser requests look like CORS failures because the proxy has no healthy backend.
- Verified production incident on 2026-05-29: API requests can fail with `QueuePool limit of size 5 overflow 10 reached` when browser polling, auth refreshes and worker load exhaust the default SQLAlchemy pool. The pool is now configurable, Compose disables the embedded worker when a standalone worker exists, worker defaults are less aggressive, and Inbox polling uses a lightweight refresh path. Production still needs PostgreSQL `max_connections` reviewed before increasing pools further.
- Verified production incident on 2026-05-29: `POST /saas/v1/auth/register` can fail with `QueuePool limit of size 10 overflow 20 reached` before creating an account when PostgreSQL connections are exhausted. Registration now releases DB before CAPTCHA/hash work, auth security-event runtime DDL is cached/serialized, and DB pool timeouts return controlled `503 database_busy`; production should still monitor pool saturation and avoid running duplicate worker modes.
- Verified production support case on 2026-05-29: a legitimate user can see `rate_limit_exceeded` after repeated wrong-password login attempts. This is expected Phase 1 protection; frontend now translates it to a friendly wait/recover-password message and backend sends `Retry-After`, but operators may still need to verify the password/reset flow or clear the lock/rate-limit records for immediate recovery.
- Verified production incident on 2026-05-29: opening `Inteligencia` can fan out many Intelligence/Trust/Ecosystem reads and trigger PostgreSQL `DeadlockDetected` when `ensure_intelligence_tables()` executes `ALTER TABLE saas_intelligence_model_registry` or `ALTER TABLE saas_intelligence_feature_values` during normal traffic. Migration `074_saas_intelligence_runtime_ddl_deadlock_repair.sql` repairs those columns at startup, `schema_readiness.py` now checks them before serving traffic, and `intelligence/service.py` caches/serializes defensive runtime DDL with a PostgreSQL advisory transaction lock.

## High

- Some SaaS social tables are not prefixed with `saas_` (`social_posts`, `social_comments`, `comment_ai_settings`). Shared schemas can collide with non-SaaS tables.
- Schema definition is split between migrations and runtime defensive `CREATE TABLE IF NOT EXISTS` in services. Inspect both before DB edits.
- Docker/API readiness now runs the schema readiness contract before serving traffic. Residual risk: future runtime-critical schema additions must update `backend/app_saas/shared/schema_readiness.py`; otherwise a newly introduced drift class may not be caught before a route uses it.
- No dedicated automated test suite was detected for SaaS during inspection; changes need targeted manual or added tests if user requests.
- Password recovery is implemented and smoke-tested locally, but production email delivery depends on valid SMTP env vars. Without SMTP, production users will not receive reset links.
- Email OTP MFA challenge enforcement is implemented in Phase 13 for tenant/admin login. TOTP/authenticator-app support is not implemented.
- Internal system notifications and Admin/Tenant user-management CRUD are implemented at repository level, but production acceptance still requires redeploy testing for SMTP delivery, notification targeting, read/pin behavior and role/status updates.
- Do not emulate future platform announcements with customer CRM conversations; extend the dedicated internal notification domain so AI/triggers/remarketing never treat platform notices as customer chats.
